Изменения в патче

1. Google OAuth
- сохранён и доработан отдельный google_oauth.py
- OAuth start без user_id в URL: GET /oauth/google/start
- OAuth disconnect без user_id в URL: DELETE /oauth/google
- callback пишет google-tokens.json в persistent secrets volume
- используется безопасный OAuth state через таблицу oauth_states

2. Единый Telegram gateway
- добавлен telegram_gateway.py
- один внешний бот принимает webhook и маршрутизирует сообщения в нужный контейнер
- контейнеры больше не слушают Telegram напрямую
- добавлены routes:
  - POST /telegram/webhook
  - POST /telegram/webhook/setup
  - GET /telegram/webhook/info
  - POST /cabinet/telegram/link/start
  - DELETE /cabinet/telegram/link
- привязка Telegram делается через deep link вида /start link_<token>

3. Новый flow Telegram
- таблица telegram_link_tokens хранит одноразовые токены привязки
- таблица telegram_links хранит связь telegram_user_id -> user_id
- по входящему сообщению backend:
  - находит пользователя
  - поднимает спящий контейнер при необходимости
  - ждёт readiness OpenClaw HTTP API
  - шлёт запрос в /v1/responses нужного контейнера
  - отправляет ответ обратно через единый бот

4. Переконфигурирован OpenClaw внутри контейнера
- entrypoint.sh больше не включает native Telegram channel
- включён HTTP Responses API: /v1/responses
- для каждого контейнера включён gateway.auth.mode=token
- добавлен отдельный GATEWAY_AUTH_TOKEN на инстанс
- bind переключён на lan, чтобы backend мог обратиться к gateway по внутреннему Docker IP

5. docker_manager.py
- добавлен gateway_token в окружение контейнера
- добавлен start_instance(user_id)
- добавлен get_container_ip(user_id)
- сохранён secrets volume user_{id}_secrets
- общий google-oauth.json монтируется read-only
- пользовательские секреты и токены переживают recreate

6. cabinet.py и cabinet.html
- в кабинет добавлено подключение/отключение Telegram
- Telegram теперь отдельная feature рядом с Google
- кнопка “Подключить Telegram” создаёт deep link и открывает бота
- disconnect удаляет связь telegram_links

7. admin/dashboard
- provisioning больше не требует Telegram token на пользователя
- dashboard отправляет Bearer token из localStorage для admin API
- список инстансов больше не отдаёт gateway_token и api_key наружу

8. schema.sql и migrate_users.sql
- в user_instances добавлен gateway_token
- добавлены таблицы:
  - telegram_link_tokens
  - telegram_links
- миграция заполняет gateway_token для существующих user_instances

Новые/важные переменные окружения backend:
- DATABASE_URL
- JWT_SECRET
- GOOGLE_OAUTH_JSON_PATH=/srv/openclaw/secrets/google-oauth.json
- GOOGLE_REDIRECT_URI=https://your-domain.example/oauth/google/callback
- ADMIN_EMAILS=admin@example.com
- TELEGRAM_BOT_TOKEN=<единый бот>
- TELEGRAM_BOT_USERNAME=YourBotNameWithoutAt
- TELEGRAM_WEBHOOK_URL=https://your-domain.example/telegram/webhook
- TELEGRAM_WEBHOOK_SECRET=<случайная строка>

Важно:
- backend должен работать на том же Docker-host, где запущены пользовательские контейнеры
- backend должен иметь доступ к Docker daemon
- /v1/responses внутри контейнера — это operator surface, поэтому gateway token хранится только на backend и в БД, не выдаётся в UI

9. Единая модель агента
- отдельная tool model удалена из Docker flow: в контейнер уходит только одна модель через LLM_MODEL
- базовая модель выбирается админом в /admin и хранится в settings
- клиентский кабинет не показывает slug/провайдера базовой модели
- клиент видит только режим: «Базовая модель» или «Особенная модель»
- «Особенная модель» требует провайдера, slug модели и пользовательский API key
- при возврате на «Базовую модель» пользовательский API key очищается и контейнер пересоздаётся с админскими настройками
- /cabinet/agent/restart теперь заново резолвит модель через sync_instance_to_admin_settings(): базовая берётся из админки, особенная — из пользовательских настроек
